# Estratégias e Princípios de Geração de Código para Linguagens de Alto Nível

Este briefing detalha os fundamentos da geração de código MIPS para uma linguagem de programação de alto nível, com foco em expressões aritméticas, estruturas de controle e a manutenção de invariantes cruciais para a integridade do programa durante o tempo de execução.

## Sumário Executivo

O processo de geração de código para linguagens que suportam inteiros, recursão e expressões condicionais exige uma abordagem sistemática para gerenciar resultados intermediários e o fluxo de controle. A análise demonstra que a geração de código baseada em uma máquina de pilha virtual é a estratégia mais robusta para lidar com expressões aninhadas, garantindo que o estado da memória seja preservado. O documento explora a distinção entre processos de tempo de compilação e tempo de execução, define invariantes para o acumulador e o ponteiro de pilha, e detalha por que tentativas simplistas de otimização usando registradores fixos falham em estruturas complexas.

## Análise de Temas Principais

### 1. Definição da Linguagem e Invariantes de Geração
A linguagem em análise é superior a uma máquina de pilha simples, incluindo:
*   **Tipos e Operações:** Inteiros, somas, subtrações e testes de igualdade.
*   **Estruturas de Controle:** Expressões `if-then-else` e chamadas de função.
*   **Ponto de Entrada:** A primeira função definida na lista de declarações é tratada como a rotina principal (`main`).

Para cada expressão $E$, o gerador de código deve cumprir dois requisitos fundamentais (invariantes):
1.  **Acumulador ($a0$):** O código gerado deve computar o valor de $E$ e deixá-lo no registrador acumulador $a0$.
2.  **Preservação da Pilha:** O estado da pilha (conteúdo e ponteiro $sp$) deve ser exatamente o mesmo antes e depois da execução do código gerado para $E$.

### 2. A Distinção entre Tempo de Compilação e Tempo de Execução
Um conceito central na arquitetura de informação do compilador é a separação temporal das operações:
*   **Tempo de Compilação (Compile-time):** Execução da função de geração de código (`Cgen(E)`). É o momento em que o compilador percorre a Árvore de Sintaxe Abstrata (AST) e emite/imprime as instruções.
*   **Tempo de Execução (Runtime):** O momento em que o programa MIPS gerado é efetivamente processado, realizando cálculos e manipulando a pilha.

### 3. Geração de Código Recursiva e Modelos (Templates)
A geração de código é descrita como um processo de **descida recursiva**. O código para operações complexas (como adição ou estruturas condicionais) funciona como um "template" com lacunas, onde o código para subexpressões é inserido recursivamente.

| Operação | Lógica de Geração de Código |
| :--- | :--- |
| **Inteiro (I)** | Carrega o valor imediato diretamente no acumulador $a0$. |
| **Soma (E1 + E2)** | Gera código para E1; armazena $a0$ na pilha; gera código para E2; recupera E1 da pilha para um registrador temporário; soma com $a0$; restaura a pilha. |
| **Subtração (E1 - E2)** | Idêntico à soma, mas utiliza a instrução de subtração (`sub`) no estágio final. |
| **Condicional (If)** | Avalia o predicado (E1 = E2) usando a pilha para gerenciar os resultados; utiliza saltos condicionais (`beq`) e incondicionais (`j`) para alternar entre as ramificações `true` e `false`. |

### 4. O Problema da Otimização via Registradores Fixos
O documento identifica uma falha crítica em tentativas de evitar o uso da pilha para armazenar valores intermediários. Ao tentar usar um registrador temporário fixo (ex: $t1$) para armazenar o resultado de E1 enquanto E2 é computado, o compilador falha em expressões aninhadas.

*   **Cenário de Erro:** Em uma expressão como `1 + (2 + 3)`, a avaliação da subexpressão interna `(2 + 3)` sobrescreveria o valor da primeira subexpressão `1` no registrador $t1$, resultando em cálculos incorretos.
*   **Conclusão:** O uso da pilha é indispensável para preservar resultados intermediários em linguagens que permitem recursão e aninhamento de expressões do mesmo tipo.

**Contexto:** Definição da invariante de pilha, essencial para garantir que chamadas de função e expressões complexas não corrompam a memória.

**Contexto:** Explicação pedagógica sobre a função `Cgen`, enfatizando que o compilador "imprime" o programa que será executado posteriormente.

**Contexto:** Descrição da natureza modular e recursiva do compilador, onde instruções fixas de MIPS emolduram o código gerado para as subexpressões.

## Insights Técnicos e Operacionais

### Implementação de Fluxo de Controle (`If-Then-Else`)
A geração de código para condicionais segue uma estrutura rigorosa para garantir a integridade dos dados:
1.  **Avaliação do Predicado:** E1 e E2 são avaliados sequencialmente. O resultado de E1 é obrigatoriamente movido para a pilha para evitar que a avaliação de E2 o destrua.
2.  **Comparação:** O valor de E1 é carregado em um registrador temporário e comparado com $a0$ (que contém E2).
3.  **Desvio:**
    *   Se iguais, o programa salta para o rótulo da ramificação verdadeira.
    *   Se diferentes, o programa executa a ramificação falsa e, em seguida, realiza um salto incondicional para o fim da estrutura para evitar a execução do código "true".
4.  **Consistência de Saída:** Ambas as ramificações (`E3` e `E4`) garantem a preservação da pilha e deixam seus respectivos resultados em $a0$, tornando o processo transparente para o restante do programa.

### Práticas Recomendadas para Geração de Código
*   **Priorizar a Pilha sobre Registradores:** Embora o uso de registradores pareça mais rápido, a pilha é a única forma segura de gerenciar profundidade arbitrária em expressões aninhadas sem análise de alocação de registradores complexa.
*   **Manter a Recursividade:** A estrutura do gerador de código deve espelhar a estrutura da Árvore de Sintaxe Abstrata (AST) da linguagem, facilitando a manutenção e a correção do compilador.
*   **Gestão de Rótulos:** Na implementação de condicionais, o compilador deve gerar rótulos únicos (como `true_branch` e `end_if`) para evitar conflitos de salto em programas com múltiplos blocos `if`.

## 1. Visão Geral da Linguagem e Gramática

A linguagem analisada é composta por inteiros, operações matemáticas e definições de funções. A estrutura básica segue estas regras:

*   **Programa:** Uma lista de declarações de funções.
*   **Declaração:** Consiste em um nome de função, uma lista de argumentos (identificadores) e um corpo (expressão).
*   **Ponto de Entrada:** A primeira função definida na lista é considerada a rotina principal (`main`).
*   **Expressões Suportadas:**
    *   Constantes inteiras.
    *   Identificadores.
    *   Estruturas condicionais (`if then else`) com testes de igualdade.
    *   Operações aritméticas: Soma (+) e Subtração (-).
    *   Chamadas de função.

**Exemplo Prático:** A linguagem é expressiva o suficiente para definir a função de Fibonacci, utilizando chamadas recursivas e expressões condicionais para determinar se o valor de entrada resulta em 0, 1 ou na soma das chamadas anteriores.

## 2. Invariantes da Geração de Código

Para que a geração de código (representada pela função `cgen(E)`) seja bem-sucedida e correta, ela deve seguir dois princípios fundamentais (invariantes):

1.  **Uso do Acumulador ($a0$):** O código gerado para qualquer expressão `E` deve computar o valor de `E` e armazenar o resultado final no registrador acumulador `$a0`.
2.  **Preservação da Pilha:** Após a execução do código de `E`, o ponteiro da pilha (`$sp`) e o conteúdo da pilha devem estar exatamente no mesmo estado em que se encontravam antes do início da avaliação da expressão.

---

## 3. Geração de Código por Tipo de Expressão

A função `cgen` opera de forma recursiva, tratando cada caso da gramática como um "template" com lacunas a serem preenchidas.

### Inteiros e Aritmética

| Tipo de Expressão | Lógica de Geração de Código | Instrução MIPS (Runtime) |
| :--- | :--- | :--- |
| **Inteiro (i)** | Carrega a constante diretamente no acumulador. | `li $a0, i` |
| **Soma (E1 + E2)** | Avalia E1, salva na pilha, avalia E2, recupera E1 da pilha para um registrador temporário, soma com o acumulador. | `cgen(E1)`, `sw $a0, 0($sp)`, `addiu $sp, $sp, -4`, `cgen(E2)`, `lw $t1, 4($sp)`, `add $a0, $t1, $a0`, `addiu $sp, $sp, 4` |
| **Subtração (E1 - E2)** | Idêntica à soma, mas substitui a instrução final de adição por uma de subtração. | `cgen(E1)`, [Push], `cgen(E2)`, [Load E1], `sub $a0, $t1, $a0`, [Pop] |

### Estruturas de Controle: If-Then-Else

A geração de código para `if E1 = E2 then E3 else E4` exige instruções de controle de fluxo:

1.  **Avaliação do Predicado:** Computa-se `E1`, armazena-se na pilha e, em seguida, computa-se `E2`.
2.  **Comparação:** O valor de `E1` é recuperado da pilha para `$t1`. A instrução `beq` (branch equal) compara `$t1` e `$a0`.
3.  **Desvios:**
    *   Se iguais: Salta para o rótulo da ramificação verdadeira (`true_branch`) para avaliar `E3`.
    *   Se diferentes: Executa a ramificação falsa (`false_branch`) para avaliar `E4`.
4.  **Finalização:** Ambas as ramificações terminam em um ponto comum (`endif`), garantindo que o valor final (de `E3` ou `E4`) permaneça no acumulador.

---

## 4. A Importância da Pilha e Riscos de Otimização

Um erro comum em compiladores iniciantes é tentar substituir a pilha por registradores fixos (como `$t1`) para armazenar resultados intermediários.

**O Problema das Expressões Aninhadas:**
Considere a expressão `(1 + 2) + 3`. Se o compilador usar apenas `$t1` para guardar o resultado de `E1` enquanto avalia `E2`:
1.  Avalia-se `1`. `$a0 = 1`.
2.  Move-se `$a0` para `$t1`. `$t1 = 1`.
3.  Avalia-se o próximo termo (`2 + 3`), que é uma nova soma.
4.  O código para `2 + 3` tentará usar `$t1` para guardar o `2`, **sobrescrevendo** o valor `1` que estava lá.
5.  O resultado final será incorreto devido ao conflito de uso do registrador.

**Conclusão:** O uso da pilha é essencial porque expressões podem ser arbitrariamente complexas e aninhadas. A pilha fornece um espaço de armazenamento recursivo que protege os valores intermediários.

## 7. Glossário de Termos Chave

*   **Acumulador ($a0$):** Registrador destinado a conter o resultado imediato de uma computação de expressão.
*   **AST (Abstract Syntax Tree):** Estrutura de árvore que representa a sintaxe do programa; a geração de código é descrita como uma descida recursiva nesta árvore.
*   **Branch Equal (`beq`):** Instrução MIPS que desvia a execução para um rótulo se os valores de dois registradores forem iguais.
*   **Cgen:** Abreviação para "Code Generation"; a função recursiva responsável por produzir código de máquina.
*   **Invariante:** Uma condição que deve permanecer verdadeira durante a execução de um trecho de código (ex: o estado da pilha antes e depois de uma expressão).
*   **Load Immediate (`li`):** Instrução que carrega um valor constante diretamente em um registrador.
*   **Template:** Modelo de código com espaços reservados onde o código gerado recursivamente para subexpressões será inserido.
